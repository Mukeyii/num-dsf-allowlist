# ADR-002 — RS256 over HS256 for JWTs and bundle signatures

- Status: Accepted
- Date: 2026-06-02
- Deciders: IMI Münster team

## Context

The portal issues JWT access tokens and temp tokens, and it signs FHIR bundles
that downstream DSF nodes consume as authoritative. With a symmetric scheme
(HS256) the same secret both signs and verifies, so every party that needs to
verify a token must hold the signing secret — and any such party can forge new
tokens. For bundles distributed across a federation that is unacceptable: a
receiver must be able to verify provenance without being handed the power to
mint bundles.

## Decision

All signing uses RS256 (asymmetric). The key pair is loaded from
`JWT_PRIVATE_KEY_BASE64` / `JWT_PUBLIC_KEY_BASE64`.

- Access and temp tokens are signed with `algorithm: 'RS256'` and verified with
  `jwt.verify(token, JWT_PUBLIC_KEY, { algorithms: ['RS256'] })` — the verify
  call pins the algorithm so an HS256-confusion downgrade is rejected
  (`backend/src/services/auth.service.ts`).
- Bundle signatures are RS256 JWTs over the bundle's SHA-256 content hash, with
  a `kid` header derived from the public-key fingerprint so verifiers can map
  signature → key and keys can rotate without invalidating old bundles
  (`backend/src/services/bundle-signing.service.ts`).
- The cryptographic admin grants of ADR-001 use the same RS256 mechanism.

## Consequences

Positive:

- Verifiers (federation peers, the admin UI) need only the public key; the
  private signing key never leaves the issuer.
- Pinning `algorithms: ['RS256']` on verify blocks the classic HS256/RS256
  algorithm-confusion attack.
- The `kid` header enables key rotation: a consumer can keep an old `kid`
  pinned until it pulls a fresh public key.

Negative:

- Asymmetric signing is more expensive than HMAC and requires managing a key
  pair and its distribution.
- Several concerns (JWTs, bundle signing, admin grants) share one key pair by
  default, concentrating risk in that single secret.
